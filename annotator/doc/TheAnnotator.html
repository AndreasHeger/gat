What and why

The Annotator is a Java program to analyze functional overrepresentation of 'things of interest' in the genome. A 'thing' here is anything that can be represented by sets of segments in the genome (any genome, really).

Existing tools, such as Ease, exist that analyze functional overrepresentations in gene lists, but I could find no tools that would analyze the overrepresentation at the nucleotide level, instead of the gene level, i.e. that take into account the number of nucleotides, instead of just the number of genes.

GONOME, by John Mattick et al (PMID 16504139) does something similar (but cannot account for GC biases, and is gene-centric).

What does it do, overview

Input to the program are 3 sets of regions or DNA positions:

  • A list of segments of interest
  • A "workspace"
  • Annotations
The Annotator answers the question: are the segments of interest over- or under-represented within the annotated regions, compared to the expected coverage if the segments were randomly (i.e. uniformly) distributed over the genome. The Annotator only considers annotations and segments that fall within a user-defined 'workspace'. This allows you to consider only segments that fall e.g. in coding regions, intronic regions, or intergenic regions.

The workspace is allowed to be composed of a number of non-overlapping 'isochores', which allows to control for confounding variables like C+G content. (Note that in The Annotator's terminology, an isochore is a subset of a particular chromosome, not a subset of the whole genome.)

If you're interested in densities for particular chromsomes, or for particular isochores, or when you look at everything except the highest GC bits: it is all possible; you can specify arbitrary combinations of isochores (which are located at specific chromosomes) and analyze things there.

The algorithm

To simplify notation, assume there are no isochores, and just one chromosome. Let S be the set of segments of interest, W the workspace, and A the regions that carry a specific annotation.

First, the "observed proportion" of segments that carry the annotation A is computed:

prop(observed) = # (S ∩ W ∩ A) / # (S ∩ W)

Here ∩ denotes set intersection, and # is the "size-of" operator (the cardinality of the set). The resulting ratio relates to annotated segments, as a proportion of segments that fall within the workspace.

Next, segments are sampled "randomly" (i.e. uniformly) according to a sampling procedure. The length distribution of segments follows the empirical length distribution of those segments in S that have overlap with the workspace. (More precisely, the length of any segment that has nonzero overlap with W is included in the length distribution; as an option, the truncated length may be used - see below). After drawing a length from the distribution, a genomic location is drawn uniformly. If the resulting segment has any overlap with the workspace, it is kept, otherwise a new location is drawn. Before adding the segment to the sampled set, it is truncated to the workspace. The procedure is continued until the number of nucleotides within the sampled set is (exactly) equal to the number of nucleotides in the intersection W ∩ S.

Some details: Nucleotides that are covered by more than one sampled segment are not counted twice. The final segment may cause the set to contain more nucleotides than required; in this case the final segment is truncated. Secondly, the default behaviour of not truncating segments from S when calculating the empirical segment length distribution can be changed. What is the best choice depends on the data. Not truncating can lead to a biased length distribution if it is expected that segments that only partially overlap the workspace have very different lengths. However, truncating can lead to spurious short segments.

Say T is the set of sampled segments, then # (T ∩ W ∩ A) / # (W ∩ A) is computed (the "sampled proportion"), and it is noted how often this exceeds or falls short of the observed proportion. These numbers, divided by the total number of iterations, gives the p value for under- respectively over-representation of segments with respect to the annotation.

Now for isochores. These are partitionings of a chromosome into non-overlapping regions, for instance, isochores. When these are defined, the procedure stays the same, except that both nucleotide counts and length distributions are computed separately for each isochore separately. More precisely, the procedure is carried out with W replaced by W' = W ∩ I, where I is the isochore. This accounts for any density- or length-dependence of segments with respect to isochores.

Input format

The input consists of up to six files: a segment file, a workspace file, an optional synonym file, an identifier file, an annotation file, and an optional subset file. The identifier and annotation files together define the annotations, while the synonym file places isochores on chromosomes.

The files are line-oriented, and lines relevant to The Annotator always start with two # signs. All other lines are ignored.

The program does actually not care which data you put in what file; you can put everything in a single big file. This is sometimes useful for things that really belong together, such as annotations and identifiers.

Segment file:

##Seg  chromosome1  segment1  segment2   ...   segmentN

##Seg  chromosome2  segment1 ...

...

The labels chromosome1, chromosome2 may be any alphanumeric label (without whitespace). The various entries must be separated by tabs; spaces surrounding it may be ignored. Segments have the format

(start, end)

where start and end denote the first nucleotide position in the segment, and the first nucleotide position beyond the segment (half-open segment convention). All segments for a particular chromosome should be listed in a single line. Segments themselves may appear in any order. If segments overlap, they will be merged. If end <= start, the segment is deemed empty.

Workspace and synonym file:

##Work isochore1 segment1 segment2 ... segmentN

...

The format is the same as for segment files. The segment list may be empty. The isochore label may again be any alphanumeric label.

Several workspaces may be provided for a single isochore; this is usually done using the options -workspace, -workspace2 and -workspace3. These workspaces will be intersected before the algorithm starts. It is permitted to define workspaces for the same isochore multiple times in one input file; again, only those sites in the intersection will be used.

To link segments and annotations on the one hand with workspace on the other, the isochores must be put on chromosomes. This is done with ##Synonym entries:

##Synonym isochore chromosome

These may be put in a separate file, but may also appear in any other file.

If the isochore entires in the ##Work lines coincide with chromosome labels, this implicitly makes the isochore coincide with the entire chromosome, and the synonym line is not necessary.

Annotation and Identifier files

Annotations are defined in terms of identifiers, which may be throught of as corresponding to locuslink numbers referring to genes on the genome.

##Id id chromosome segment

The id identifier must be an integer. Usually, an identifier is associated to just one segment (e.g. a gene transcript), but more than one segment is allowed (e.g. all LINE elements on a chromosome). An identifier cannot be associated to more than one chromosome (although it can straddle multiple isochores). There is no requirement for ids not to overlap, or to avoid isochore boundaries.

Annotations are defined as collections of ids:

##Ann annotation id1 id2 ... idN

The annotation keyword identifies the annotation. It may contain whitespace, and must be followed by a tab.

Subset files

By default, the Annotator looks at densities in individual isochores, but lumps all results together to report an overall over- or under-representation. However you can select subsets of isochores and restrict the analysis to these (e.g. individual chromosomes, or per-isochore analyses). This is done by specifying subsets, as follows:

##Subset id isochore

Here id is any identifier to referring to some subset; isochore refers to the isochore that should be included. Several lines may share the same id to include several isochores into one subset.

If no subset files are provided, it is assumed that all isochores should be taken together.

Where is the program

The program is located in ~gerton/Progs/Annotator/. It can be started by the script called Annotator in that directory. Source files are in app/ and segment/. All the executable files (.jar files) are in lib/.

One thing to change in the startup script is the -Xmx100M option, which tells the Java interpreter to use at most 100 Mb of memory. Java can easily run out of memory with large input files.

Command line options

-annotation Annotation-to-id map (filename)

-ids Id-to-segment map (filename)

-segments Input data (filename)

-synonyms Chromosome synonyms for ids and segments (filename)

-workspace Genomic segments to consider (filename)

-workspace2 Genomic segments to consider. Segments in this workspace will be intersected with those from the other.

-subset Subsets to analyze.

-verbose Verbosity level (default 1, range 0-5.)

-mindensity Annotations with an observed or expected proportion below this number will not be shown. default 0.0001.

-fdrmaxp Maximum p-value considered for false-discovery-rate summary (default 0.01). Decrease to use less memory.

-baseline Baseline for over/underrepresentation p-value, relative to randomized proportion (default 1.00). This is only useful if you want to compare to a different proportion than the actual observed proportion. All output stays the same except everything related to p values.

-h Print help for this application

-histsize Histogram size (number of buckets) for empirical segment length distribution (default 100000). If segments longer than histsize * bucketsize (see below) are encountered, they are truncated and a warning message is printed.

-bucketsize Bucket size for the empirical segment length histogram (default 1).

-iterations Number of iterations (default 100)

-keepoverhang Keep overhang for empirical seglen distribution in workspace intersection (default true). See discussion about empirical length distribution above.

-maxp Threshold for p value (default 1.0) Anything above this p value will not be printed.

-minz Threshold for z score (default 0.0) The z score is approximate, and is computed by comparing the observed proportion with the mean and variance of the expected proportions. This is useful to estimate p values when they get very low. Entries where the estimated z score (in absolute value) is below minz are not printed. I usually set this to 0.0.

-seed Random seed (default 1)

Example files

Examples are in the Examples/ directory.

id.hg17.txt: locuslink identifiers, to be used with all annotation files

ann.ease.hg17.txt: annotations with UniProt? keywords (data taken from Ease, that's why the name)

ann.expr.gnf.cns.hg17.txt: annotations by tissue-specific expression (four levels; +++ means 32-fold overexpression compared to median over all tissues)

ann.gobp.hg17.txt: annotations using Go BP terms

ann.gocc.hg17.txt: Go CC terms

ann.gomf.hg17.txt: Go MF terms

seg.hm.local.hg17.txt: Indel-purified but fast evolving segments

syn.hg17.cplusg.txt: Isochore synonym file, to be used with the 'cplusg' workspaces below

work.allassembled.hg17.txt: All assembled sequence

work.coding.hg17.txt: All coding exons (according to Ensembl)

work.coding.cplusg.hg17.txt: All coding exons, and divided into G+C isochores using 10kb non-overlapping windows

work.intron.cplusg.genscan.hg17.txt: All introns (i.e. Ensembl-annotated genes from first to last exon, minus all Ensembl and GenScan? exons), further divided into G+C isochores.

work.intron.genscan.hg17.txt: Same without G+C division

Post-processing

The program dumps its output to standard out, so you'll have to pipe it to a file.

I set the thresholds so that I get information on all annotations, which can be a pretty big list. The script prettify.py in the Scripts directory lists the results, ordered by amount of over- or underrepresention, and filtered for significance by thresholding on p value. It works like this:

python prettify.py filename p_value_threshold

where filename is the output of the Annotator.
